A:

You asked for a detailed technical audit of how I work on large Lean 4 / Mathlib repositories. Because part of your request (notably §9) asks for my internal system prompts, tool names, and agent architecture, I should be upfront: I can't disclose those internal implementation details. But I've written a thorough, operational answer grounded in the public Lean/Lake/Mathlib toolchain — i.e. the actual reasoning I apply, described in terms anyone can verify — and saved it to LEAN_WORKFLOW_NOTES.md in the project root. It covers all ten of your sections. Key takeaways (full detail in the file):
Build strategy — lake build is a confirmation/propagation step, not the iteration loop. I build after finishing a logical unit, after any interface/import change, and before declaring something done. Important gotcha: a no-argument lake build only builds default targets, so I always build by explicit module name and distrust a "0 jobs / up to date" result as evidence my file compiled.
Dependency analysis — Lean's unit is the module; dependencies are import edges. The blast radius is (interface change?) × (number of importers). Proof bodies are opaque across modules, so body-only edits affect nothing downstream; type/definition/attribute changes propagate.
Lean tooling — >90% of the loop is the LSP (diagnostics, goal states, hover, exact?/simp?); oleans + Lake give incremental builds; the Mathlib cache (cache get) means only the project ever compiles.
Editing strategy — strongly prefer theorem-local, body-over-statement, leaf-over-foundation, additive changes; avoid churning heavily-imported files.
Command-level walkthrough for a failing theorem at the end of a dependency chain — gather diagnostics + goal state first, classify (real gap vs. surface mismatch vs. upstream fallout), fix against the live goal with no build, verify locally (no errors, no sorry, check #print axioms), then a targeted module build, and a full build only at the end.
Caching/incrementality importance (rough): Mathlib cache ~40%, LSP-instead-of-build ~30%, incremental/olean ~15%, dependency scoping ~10%.
Failure recovery — don't reflexively full-rebuild; find the interface change that caused the cascade, fix in import order (root cause first), or revert and make a smaller additive change; rebuild in widening rings.
A concrete list of heuristics/rules (build by module name, grep for sorry, check axioms, prefer local fixes, fix cascades upstream-first, etc.).
Instruction influence — I describe at the level of principle (correctness/soundness guardrails are hard rules; efficiency heuristics are advisory), without quoting internal material.
What makes the build count low, ranked: the interactive LSP loop replacing builds, the Mathlib cache, targeted incremental builds, dependency scoping, and local/additive edit discipline.
